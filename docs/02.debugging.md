# Software Configuration & Debugging

There are many code snippets shared online, but I found that most of them didn't work for my specific setup. After several nights of troubleshooting, I identified the root causes and finalized a configuration that works perfectly for my hardware. 

**Note:** Hardware varies from person to person. What works for others might not work for you. Please use the following as a reference and adjust based on your specific needs.

---

## 1. The Key Solution: Network Performance Mode
First thing first, the most critical discovery for my hardware was the `network: enable_high_performance: false` setting. 

In recent ESPHome versions, this is often enabled by default when using media player components. However, I constantly encountered "decoding" and "ring buffer" errors. On my device, the "High Performance" mode consumed excessive SRAM, leaving insufficient space for the audio ring buffer, which caused stuttering or total decoder failure.

**By explicitly setting `enable_high_performance: false`, my media player finally began to play smoothly.**

## 2. 🛠Implementation Approaches & Debugging
During development, I experimented with three distinct software architectures to find the best balance between audio stability, feature set, and resource management.

### 1. Standard Dual-Channel Speaker Architecture
This approach uses the standard ESPHome `Speaker Audio Media Player` components configured for dual-channel output.

* **Performance Observations:**
    Despite extensive `sdkconfig` optimizations for SRAM/PSRAM and various `buffer_duration` adjustments, I could not achieve 100% stutter-free playback for **FLAC** files; occasional stutter persisted. However, **MP3** playback was flawlessly stable.
* **Optimal Configuration:** See [`speaker_audio_media_player.yaml`](../yamls/speaker_audio_media_player.yaml).
* **Usage Notes:**
    * **Integration:** Must be added to Home Assistant (HA) via the ESPHome integration first.
    * **Music Assistant (MA) Setup:** To use this in MA, navigate to *Settings -> Player Providers -> Home Assistant MediaPlayers* and manually add the entity.

### 2. Sendspin-Optimized Speaker Source
This is a "lean" configuration designed specifically for the `Sendspin` protocol, utilizing a single `media_source`. I removed the `mixer` component and retained only one `resampler` for a more direct pipeline.

* **Performance Observations:**
    Interestingly, this configuration uses **more PSRAM** than the dual-channel approach but **reduces SRAM overhead**. Crucially, **FLAC playback was completely smooth** with no stuttering issues.
* **Optimal Configuration:** See [`simple_sendspin_media_player.yaml`](../yamls/simple_sendspin_media_player.yaml).
* **Usage Notes:**
    * **MA Native Support:** MA recognizes `Sendspin` as a native protocol, which is more robust than the standard HA player integration. MA will auto-discover the player upon boot.
    * **Entity Behavior:** This setup does *not* create a `media_player` entity directly under the HA ESPHome device (sensors remain normal). You can, however, sync it back to HA as a media player entity through MA settings.
    * **Optimization Tip:** MA defaults to a **48,000Hz** sampling rate for Sendspin. Setting your ESPHome `sample_rate` to `48000` bypasses the need for the `resampler` speaker, saving CPU cycles.

### 3. Hybrid Pipeline (Voice-PE Style)
Inspired by the `home-assistant-voice-pe` configuration, this setup uses `external_components` (custom `http_request`, `media_source`, and `sendspin`) to build separate **Announcement** and **Media** pipelines.

* **Performance Observations:**
    This is the most feature-rich version, allowing the player to act as a native Sendspin device in MA while simultaneously exposing a `media_player` entity to HA for TTS and notifications. Similar to the first approach, **FLAC** files occasionally struggled with bandwidth/buffer issues, so **MP3** remains the recommended format for total reliability.
* **Optimal Configuration:** See [`speaker_source_sendspins_media_player.yaml`](../yamls/speaker_source_sendspins_media_player.yaml).

## 3. Configuration Breakdown
Here are some specific parts of my code that you may want to understand or adapt:

### PlatformIO & PSRAM Support
The default `esp32-s3-devkitc-1` board definition in ESPHome does not include PSRAM support. To ensure the **N16R8** module is correctly recognized with its 8MB Octal PSRAM, I explicitly declared the `platformio_options`. This is essential for the board to utilize the extended memory.

```
  platformio_options: 
    board_build.f_flash: 80000000L
    board_build.f_cpu: 240000000L
    board_build.flash_mode: qio
    board_build.psram_type: opi
    board_build.arduino.memory_type: qio_opi
    board_upload.maximum_size: 16777216
    board_upload.speed: 921600
    build_flags: 
      - -DBOARD_HAS_PSRAM
```

### SDKConfig Optimizations
Under the `esp32` section, I applied several `sdkconfig_options` to free up as much SRAM as possible (e.g., disabling Bluetooth). 
* **Disclaimer:** I am still exploring the full impact of some of these settings. Please use them with caution.

```
    sdkconfig_options:
      CONFIG_BT_ENABLED: "n"
      CONFIG_ESP_WIFI_TASK_CORE_ID: "0"
      # Configs to improve data cache, these configs consume about 65KB SRAM
      CONFIG_ESP32S3_DATA_CACHE_64KB: "y"
      CONFIG_ESP32S3_DATA_CACHE_LINE_64B: "y"
      CONFIG_ESP32S3_INSTRUCTION_CACHE_32KB: "y"
      # Configs to optimize PSRAM usage
      CONFIG_SPIRAM_RODATA: "y"
      CONFIG_SPIRAM_FETCH_INSTRUCTIONS: "y"
      CONFIG_SPIRAM_TRY_ALLOCATE_WIFI_LWIP: "y"
      CONFIG_SPIRAM_ALLOW_BSS_SEG_EXTERNAL_MEMORY: "y"
      CONFIG_MBEDTLS_EXTERNAL_MEM_ALLOC: "y"
```

### Add-on Packages (Addons)
The content in the `packages` section (like `monitor_memory.yaml`) does not affect the core functionality. You can add or remove these as needed:
* **Deep Sleep:** I implemented a feature that triggers deep sleep when the headphone jack detection pin is high (no plug inserted). If you didn't solder the detection wire, remove this package.
* **Sensors & Monitoring:** The sensors included in the YAML are for debugging purposes to monitor SRAM and PSRAM usage in real-time. You can remove them once your setup is stable.
