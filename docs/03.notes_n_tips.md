# 💡 Notes & Tips

_Detailed observations and technical solutions discovered during the development and debugging of the ESP32-S3 Media Player._

---

## 1. Pin Pull-up Workaround (Crucial)
During debugging, I found that setting `pullup: true` within the `deep_sleep` component didn't always take effect, leaving the pin in a floating state.
```
binary_sensor:
  - platform: gpio
    pin:
      number: GPIO11
      mode: INPUT_PULLUP # this PULLUP setting make it work
      allow_other_uses: true
    #use_interrupt: false
    name: "Plug Detection"
    id: plug_detection
deep_sleep:
  id: ds_config
  wakeup_pin:
    - pin:
        number: GPIO11
        # mode: INPUT_PULLUP --> the PULLUP setting here doesn't work
        allow_other_uses: true
      wakeup_pin_mode: IGNORE
```
* **Solution:** Re-declare the same pin as a `binary_sensor` and set the pull-up there. This forces the ESP32-S3 to correctly initialize the internal resistor.

## 2. Hardware "Gotchas"
* **DAC Silkscreen Error:** My PCM5102A board had inverted silkscreen labels for the H/L jumpers. Always verify continuity with a multimeter to confirm GND (L) before bridging pads.
* **Orientation:** Double-check the orientation of the wires relative to the board's front and back before soldering. It’s easy to mirror them by mistake.
* **Pin Conflict:** If using my enclosure design, **avoid using GPIO 3 and 8**, as these pins align with the internal support brackets. Additionally, **avoid GPIO 46**, as it is a strapping pin with specific hardware constraints.

## 3. Mute Control (XSMT) Simplification
In my initial build, I used a dedicated wire for software-based mute control. However, after extensive testing, I found the output quality of the PCM5102A to be excellent and free of significant pop noise. To simplify the build, I removed the control wire and used a solder bridge to permanently connect **XSMT (Pin 3)** to **High (H)**.

## 4. Network Performance vs. PSRAM Settings
I discovered an interesting relationship between the `network` and `psram` configuration flags in ESPHome. These settings significantly impact memory allocation and TCP window sizes:

* **Case A: High Performance Enabled**
  * **Settings:** `network: enable_high_performance: true` (or default) + `psram: ignore_not_found: false`.
  * **Log Snippet:**
    ```text
    INFO Applying high-performance lwip settings (PSRAM guaranteed): 512KB TCP windows, 512 mailbox sizes
    INFO Applying high-performance WiFi settings (PSRAM guaranteed): 512 RX buffers, 32 TX buffers
    ```
  * **Result:** This is the most aggressive setting for data throughput, requiring guaranteed PSRAM availability.

* **Case B: User Overridden (Partial)**
  * **Settings:** `network: enable_high_performance: false` + `psram: ignore_not_found: false`.
  * **Log Snippet:**
    ```text
    INFO High performance networking disabled by user configuration (overriding component request)
    INFO Applying high-performance WiFi settings (PSRAM guaranteed): 512 RX buffers, 32 TX buffers
    ```
  * **Result:** TCP buffers are restricted, but the WiFi driver still utilizes high-performance RX buffers because PSRAM is guaranteed.

* **Case C: Optimized/Conservative Mode**
  * **Settings:** `network: enable_high_performance: false` + `psram: ignore_not_found: true` (or default).
  * **Log Snippet:**
    ```text
    INFO Applying optimized lwip settings: 65KB TCP windows, 64 mailbox sizes
    INFO Applying optimized WiFi settings: 64 RX buffers, 64 TX buffers
    ```
  * **Result:** ESPHome scales down to conservative "Optimized" settings, significantly reducing memory footprint.

**Technical Discovery:** The `enable_high_performance` flag primarily affects the LwIP (TCP/IP stack) send/receive windows, whereas the "PSRAM guaranteed" status (controlled by `ignore_not_found: false`) is the prerequisite for the system to scale up the WiFi driver's internal RX/TX buffers.

## 5. Pipeline Format Priority
An unusual behavior was observed when both `announcement_pipeline` and `media_pipeline` are defined. If the audio formats (e.g., MP3 vs. FLAC) differ between the two pipelines, Home Assistant or Music Assistant defaults to the format defined in the **announcement pipeline**, even when streaming regular media. To ensure consistency and avoid unexpected transcoding, it is recommended to keep the format settings identical across both pipelines.
